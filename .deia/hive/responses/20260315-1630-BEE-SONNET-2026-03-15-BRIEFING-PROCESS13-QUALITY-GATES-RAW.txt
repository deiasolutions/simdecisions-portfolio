- Current flow: `spec file → dispatch.py (calls bee) → baseline → dispatch → verify`

**Two interpretations:**

- Pro: Catches broken repo state before wasting bee cost
- Pro: Aligns with "validate the plan" (Act 1)
- Con: Not the bee's fault if repo is broken — might flag specs as NEEDS_DAVE when the issue is environmental
- When: In `process_spec()`, after loading spec but before `_capture_baseline`

**My recommendation:** **Option A (before dispatch)** with graceful degradation. Rationale:
1. The spec explicitly says "spec validation → build → test" (sequential, pre-dispatch)
2. Catching a broken build BEFORE spending $0.50-$5 on a bee run is cost-efficient
3. If build check fails, we skip dispatch entirely and flag NEEDS_DAVE with clear error
4. This matches the "validate the plan" gate in P-04 Act 1
