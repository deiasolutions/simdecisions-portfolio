## What Was Done

### Problem Identified

The Canvas IR generation was experiencing routing issues due to duplicate processing of IR responses:

1. **Envelope routing** (`routeEnvelope`) was correctly parsing and routing `to_ir` to canvas
2. **IR block extraction** (`extractIRBlocks`) was ALSO running on `envelope.to_user` content
3. This caused either duplicate routing or responses appearing in wrong panes

### Root Cause

In `useTerminal.ts`, the code flow was:
1. Parse envelope and route slots (lines 667-684)
2. THEN extract IR blocks from `displayContent` (line 706)

This meant IR blocks were being extracted and routed TWICE when the LLM used proper envelope format.

### Solution Implemented

**Key insight:** Only extract IR blocks manually when envelope parsing FAILED (fallback to plain text with fenced JSON blocks).

Changes made:

1. **Track envelope parse errors** (lines 658, 677, 685):
   - Added `envelopeParseError` variable to capture `parseError` from envelope parsing
   - This indicates whether the LLM used proper envelope format or fell back to plain text

2. **Conditional IR extraction** (line 708):
   - Changed condition from `if (isIRMode && irTargetId)`
   - To: `if (isIRMode && irTargetId && envelopeParseError)`
   - Only extracts IR blocks in fallback mode (when envelope parsing failed)

3. **New envelope-success path** (lines 745-762):
   - Added `else if (isIRMode && irTargetId && !envelopeParseError)` branch
   - Handles IR mode when envelope WAS successfully parsed
   - Sends `to_user` content to chat pane
   - Shows metrics in terminal
   - Skips IR extraction (already routed by `routeEnvelope`)

4. **Error routing fix** (line 799):
   - Changed condition from `if (isChatMode && chatTargetId)`
   - To: `if ((isChatMode || isIRMode) && chatTargetId && bus && nodeId)`
   - Ensures errors in IR mode are visible in chat pane

### Flow After Fix

**Envelope mode (proper format):**
1. LLM returns: `{ "to_user": "...", "to_ir": {...} }`
2. `routeEnvelope` parses and sends `to_ir` to canvas-editor ✓
3. `displayContent` = `envelope.to_user`
4. NEW path: Send `to_user` to chat pane, show metrics in terminal
5. Skip `extractIRBlocks` (envelope already handled IR routing)

**Fallback mode (plain text with fenced blocks):**
1. LLM returns: `"Here's the flow:\n\n```json\n{...}\n```"`
2. `routeEnvelope` gets parse error (not valid JSON envelope)
3. `envelopeParseError` = "Response was plain text"
4. Extract IR blocks from plain text using `extractIRBlocks`
5. Route chat text and IR blocks separately

## Technical Notes

### Why the Fix Works

**Before:** The code assumed IR mode meant "always extract JSON blocks from content"

**After:** The code now distinguishes:
- **Envelope mode:** LLM formats response properly → routing handled by `routeEnvelope`
- **Fallback mode:** LLM sends plain text → manually extract and route IR blocks

### Backward Compatibility

The fix maintains full backward compatibility:
- Old LLMs that don't use envelopes still work (fallback path)
- New LLMs using envelope format work correctly (envelope path)
- Error handling improved for both modes

### Performance Impact

Minimal performance impact:
- One additional boolean check per response
- No extra bus messages
- Reduced duplicate routing (slight performance improvement)
