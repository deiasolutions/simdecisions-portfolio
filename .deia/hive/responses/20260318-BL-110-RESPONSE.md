## Analysis Findings

### Current Status Systems (5 distinct models)

1. **Kanban Columns** (`VALID_KANBAN_COLUMNS` in `hivenode/inventory/store.py:108`)
   - icebox, backlog, in_progress, review, done
   - Used by: backlog items in kanban board

2. **Dev Cycle Stages** (`VALID_STAGES` in `hivenode/inventory/store.py:109`)
   - SPEC, IR, VAL, BUILD, TEST
   - Used by: stage_log_table tracking item progress

3. **Stage Statuses** (`VALID_STAGE_STATUSES` in `hivenode/inventory/store.py:110`)
   - done, active, pending, failed, blocked
   - Used by: stage_log_table for each stage

4. **Feature Inventory Statuses** (`VALID_STATUSES` in `hivenode/inventory/store.py:102`)
   - BUILT, SPECCED, BROKEN, REMOVED
   - Used by: features_table

5. **Bug Statuses** (`VALID_BUG_STATUSES` in `hivenode/inventory/store.py:106`)
   - OPEN, ASSIGNED, FIXED, WONTFIX
   - Used by: bugs_table

### The Problem

The spec asks to "align the status systems across kanban, dev cycle, and inventory so they use a single unified status model."

However, these systems serve **different purposes**:

- **Kanban columns** = spatial location on a board (where the card is)
- **Dev cycle stages** = temporal progression through workflow phases (SPEC → IR → VAL → BUILD → TEST)
- **Stage statuses** = state of each stage (active, done, failed, etc.)
- **Feature statuses** = build health (BUILT, BROKEN, etc.)
- **Bug statuses** = resolution state (OPEN, FIXED, etc.)

### Strategic Questions (Q88N decision required)

1. **Which system is canonical?** Should kanban columns map TO dev stages, or dev stages map TO kanban columns?
2. **What is the unified model?** A single enum that covers all use cases, or a mapping layer between systems?
3. **How do we handle stage_log temporal tracking?** The `stage_log_table` tracks progression through stages over time. Kanban columns are spatial. How do these relate?
4. **What about feature health vs. item location?** A feature can be BUILT (healthy) but sitting in the "review" kanban column. Are these orthogonal dimensions?
5. **Status transitions:** What are the valid transitions in the unified model? Can an item go from "in_progress" back to "backlog"?

### Architectural Impact

Unifying these systems affects:

- **Database schema** (backlog_table, bugs_table, stage_log_table, features_table)
- **API contracts** (kanban routes, build monitor routes)
- **Frontend state management** (kanban pane, apps home)
- **Dev cycle tracking** (PROCESS-LIBRARY-V2.md P-01 lifecycle)
- **Inventory reporting** (FEATURE-INVENTORY.md generation)

### Recommended Approach

**Option A: Orthogonal Dimensions (no unification)**
- Keep kanban columns separate from dev stages
- Add a `dev_stage` column to backlog_table (SPEC, IR, VAL, BUILD, TEST)
- Add a `health_status` column to backlog_table (BUILT, BROKEN, etc.)
- Kanban column = where it is, dev_stage = workflow phase, health_status = build state

**Option B: Unified Linear Model**
- Define a single status enum with all states: draft, spec_done, ir_done, val_done, build_done, test_done, review, shipped, broken, archived
- Map all existing systems to this model
- Deprecate separate kanban_column, stage, status fields

**Option C: Mapping Layer**
- Keep all 5 systems as-is
- Create a status translation service that maps between them
- Define rules like: "in_progress kanban → BUILD stage + active status"
