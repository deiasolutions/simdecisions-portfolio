## What Was Done

**Root cause identified:**
1. E2E test timeout was too short (max_wait=10s) for server startup
2. httpx.get() timeout was also too short (2.0s), causing premature failures
3. Server startup actually takes 10-15 seconds due to:
   - Database initialization (SQLite tables for phase_ir, inventory, efemera, etc.)
   - Sync worker initialization (even though it fails gracefully, it adds delay)
   - RAG and repo indexer initialization

**Debugging process:**
1. Created standalone debug scripts to capture actual server startup behavior
2. Discovered stdout buffering was hiding server output
3. Used `PYTHONUNBUFFERED=1` and `-u` flag to see real-time output
4. Confirmed server starts successfully after ~10.75 seconds
5. Identified that health check timeout (2s) was shorter than startup time

**Secondary issue fixed:**
- Node announcement logic was incorrect: local mode nodes should NOT attempt to announce to cloud hub
- Changed `if settings.mode in ["local", "remote"]` to `if settings.mode == "remote"`
- This prevents unnecessary network calls during local E2E testing

**Fixes applied:**
1. **main.py:** Fixed node announcement to only happen in `remote` mode (not `local`)
2. **test_e2e.py:** Increased timeouts to accommodate actual server startup time:
   - max_wait: 10.0s → 20.0s
   - httpx timeout: 2.0s → 5.0s
   - sleep interval: 0.2s → 0.5s

## Issues / Follow-ups

### Fixed issues
1. ✅ E2E server startup timeout (BUG-043) - FIXED
2. ✅ Node announcement logic incorrect for local mode - FIXED

### Discovered issues (not blocking)
1. **Sync worker errors during startup:** The sync worker throws errors about missing railway_object_storage config, but these are caught gracefully. Not blocking, but could be cleaner.
   - Error: `railway_object_storage adapter requires 'endpoint_url' and 'bucket'`
   - Occurs during: `await sync_engine.sync("cloud", "home")`
   - Impact: None (errors are caught, server continues)
   - Recommendation: Skip cloud sync entirely in local mode (BL-203 may already address this)

2. **Stdout buffering on Windows:** During investigation, discovered that subprocess stdout is heavily buffered on Windows, making debugging difficult. Used `PYTHONUNBUFFERED=1` environment variable and `-u` Python flag to work around this.
   - Impact: Made debugging harder, but no runtime impact
   - Recommendation: Document this in E2E test comments for future debugging

### Edge cases
1. **Very slow machines:** The 20s timeout should be sufficient for most cases, but on extremely slow CI machines, startup could theoretically take longer. Monitor CI runs.
2. **Network delays:** If cloud sync is enabled via config.yml, startup could be slower. Current fix assumes sync is disabled (default).

### Recommended next tasks
1. **BL-203 (Heartbeat split):** May further improve startup time by deferring non-critical initialization
2. **Consider lazy initialization:** RAG engine and repo indexer could be initialized on first use rather than at startup
3. **Add startup logging:** Add INFO-level logs for each initialization phase to help debug future startup issues

### Not related to BUG-042
As hypothesized in the task brief, this issue was **independent of BUG-042** (BUS signature change). BUG-042 affects governance/dispositions/heartbeat tests that directly import the BUS class. The E2E server startup issue was purely a timeout configuration problem.
