## What Was Done

### Problem Fixed
When `dispatch.py` dispatches a bee via `ClaudeCodeCLIAdapter`, the bee spawns child processes (vitest, vite, node subprocesses). When the bee finishes, `adapter.stop_session()` kills the Claude Code process but does not kill its child process tree. Result: orphaned node.exe processes accumulate, each consuming 1-7 GB RAM.

This was a **production-critical memory leak** on the Q88N workstation.

### Solution Implemented

1. **Added `_kill_child_processes()` function to dispatch.py** (lines 271-312)
   - Uses psutil to find and kill all child processes recursively
   - Gets PID from `adapter.process.process.pid`
   - Best-effort: catches and logs exceptions, never crashes dispatch
   - Handles missing psutil gracefully with warning message
   - Handles NoSuchProcess, AccessDenied exceptions silently
   - Returns count of killed processes for logging

2. **Integrated cleanup into `dispatch_bee()` workflow** (line 391)
   - Added call to `_kill_child_processes(adapter)` immediately after `adapter.stop_session()`
   - This is a secondary cleanup layer (primary is ClaudeCodeProcess._kill_tree())
   - Acts as safety net to catch any stragglers

3. **psutil dependency** (already present)
   - Verified psutil>=5.0 already in pyproject.toml line 22
   - No dependency changes needed

4. **Comprehensive test coverage** (15 tests, all passing)
   - `test_child_process_cleanup.py`: Unit tests for cleanup logic
     - Test PID exposure from adapter
     - Test stop_session calls terminate
     - Test psutil kill logic with mocks
     - Test error handling (missing psutil, NoSuchProcess, AccessDenied)

   - `test_dispatch_cleanup_integration.py`: Integration tests

### How It Works

1. Bee dispatched → `adapter.start_session()` spawns Claude Code subprocess
2. Bee runs, spawns children (vitest, vite, node, etc.)
3. Bee finishes → `adapter.stop_session()` kills main Claude Code process
4. **NEW:** `_kill_child_processes(adapter)` finds orphaned children via psutil
5. Each child process is killed via `child.kill()`
6. Log message shows count: `[DISPATCH] Cleaned up N orphaned child processes`

### Edge Cases Handled

- **No psutil installed:** Warning logged, returns 0, dispatch continues
- **Process already gone:** NoSuchProcess caught, returns 0
- **Permission denied:** AccessDenied caught silently (best effort)
- **Some children fail to kill:** Loop continues, kills remaining children
- **Adapter has no PID:** Returns 0 immediately (process never started or already cleaned)
- **Process attribute missing:** Safe attribute access, returns 0

### Why This Works on Windows

The `ClaudeCodeProcess._kill_tree()` already uses `taskkill /F /T /PID <pid>` on Windows, which should kill the tree. However, in practice, some child processes survive. The psutil-based cleanup is a **secondary safeguard** that uses a different mechanism (recursive `children()` + `kill()`) to catch stragglers.

This dual-layer approach ensures maximum cleanup reliability:
- Layer 1: OS-native tree kill (taskkill on Windows, killpg on Unix)
- Layer 2: psutil recursive scan and kill
