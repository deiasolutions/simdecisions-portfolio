## Root Cause Analysis

### Issue 1: **Race Condition in `is_running` Check**
The `is_running` property checks `self._running and self._task is not None and not self._task.done()`, but:
- `self._running = True` is set BEFORE the task is created (line 95)
- If the task fails immediately during `asyncio.to_thread()` creation, `is_running` returns `False` even though `_running=True`

### Issue 2: **Silent Failure on Early Return**
In `queue_bridge.py` lines 76-93, if config/queue_dir don't exist or module load fails, the method returns early WITHOUT setting `app.state.queue_bridge`. The `/build/queue-runner-status` endpoint then reports `running: false` forever.

### Issue 3: **`queue-wake` Restart Logic Bug**
In `build_monitor.py:625-650`, the restart logic has a flaw:
- It checks `bridge.is_running` first
- If runner died silently (task done, `_running=False`), it tries to restart
- BUT: it calls `bridge.start()` which checks `_config_path.exists()` again
- If config doesn't exist (or any early-return condition), restart fails silently

### Issue 4: **No Structured Logging for Lifecycle Events**
The bridge logs start/stop/errors, but NOT:
- "Config missing: {path}"
- "Queue dir missing: {path}"
- "Module load succeeded"
- "Task created successfully"
- "Task failed during creation"

### Issue 5: **Auto-Restart Race Condition**
In `_on_task_done` (lines 138-149), auto-restart uses `loop.call_later()` with a lambda that calls `asyncio.ensure_future(self.start())`. This can race with manual restarts from `/build/queue-wake`.
