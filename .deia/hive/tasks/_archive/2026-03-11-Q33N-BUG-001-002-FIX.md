## BUG-001: Text-pane not rendering markdown properly

**Severity:** P1
**Component:** text-pane (SDEditor)

### Broken Behavior
The text-pane receives content like `**You:** hello` and `**claude-sonnet-4-5-20250929:** response text` via `terminal:text-patch` bus messages. The SDEditor's `renderMarkdown()` function does NOT handle inline markdown formatting. It renders each line as a `<div>` but treats `**bold**`, `*italic*`, `[links](url)`, and inline `code` as literal text. The user sees raw asterisks instead of bold text.

### Expected Behavior
- `**text**` renders as **bold**
- `*text*` renders as *italic*
- `[text](url)` renders as a clickable link
- `` `code` `` renders with code styling
- `**You:**` and `**model-name:**` render as bold labels, not raw `**You:**`

### Files Involved
- `browser/src/primitives/text-pane/SDEditor.tsx` — `renderMarkdown()` function (lines 43-134). This is the renderer that needs inline formatting support.
- `browser/src/primitives/text-pane/sd-editor.css` — CSS classes for rendered elements (`sde-md-*`). May need new classes for bold/italic/code/link.
- `browser/src/primitives/text-pane/__tests__/` — Tests for the text-pane. New tests needed for inline markdown rendering.

### Constraints
- Do NOT replace `renderMarkdown()` with a third-party library. It must remain a custom renderer.
- The renderer must be fault-tolerant: malformed markdown renders as plain text, never crashes.
- CSS: `var(--sd-*)` variables only. No hex, no rgb(), no named colors.
- The function is already 90 lines. Keep it under 150 lines after changes; extract a helper for inline formatting if needed.

### Acceptance Criteria
- [ ] `**bold**` renders bold
- [ ] `*italic*` renders italic
- [ ] `` `inline code` `` renders with code background
- [ ] `[text](url)` renders as clickable link
- [ ] Mixed inline: `**You:** hello *world*` renders correctly
- [ ] Malformed markdown (unmatched `**`, etc.) renders as plain text
- [ ] Tests cover all inline formatting cases
- [ ] All existing text-pane tests still pass

## BUG-002: Terminal mini-display echoing full chat content instead of status only

**Severity:** P1
**Component:** terminal

### Broken Behavior
In chat mode (`routeTarget: 'ai'`, used by chat.egg.md), the terminal pane shows:
1. Full LLM response text in `<ResponseContent>` blocks (duplicating what's in the text-pane)
2. The `hive> thinking...` spinner during API calls (this is fine)
3. User input entries are hidden (via `hidden: true` flag) — this part works

The terminal in chat mode should be a **status display**, not a duplicate chat window. The full response text clutters it.

### Expected Behavior
In chat mode (`routeTarget === 'ai'`):
- Terminal shows: spinner while waiting, metrics after response (clock/cost/carbon), system messages, errors
- Terminal does NOT show: full LLM response text (that's in the text-pane)
- User input entries remain hidden (already implemented via `hidden` flag)
- In non-chat mode (`routeTarget === 'shell'`): behavior unchanged — full response renders in terminal as before

### Files Involved
- `browser/src/primitives/terminal/useTerminal.ts` — `handleSubmit()` lines 334-350. In the `isChatMode` branch (line 334), the response entry is added with full `displayContent`. It should instead add a metrics-only entry.
- `browser/src/primitives/terminal/types.ts` — `TerminalEntry` union type (line 19). May need a new entry type like `{ type: 'metrics'; metrics: TerminalMetrics }` or the response entry needs a way to suppress content rendering.
- `browser/src/primitives/terminal/TerminalOutput.tsx` — `TerminalLine` component (line 74). Renders response entries with full `<ResponseContent>`. Needs to handle metrics-only entries.
- `browser/src/primitives/terminal/__tests__/` — Tests for terminal output rendering.

### Constraints
- The response entry MUST still be stored in `entries` with full content for conversation history building (lines 268-276 of useTerminal.ts build LLM context from entries).
- The `hidden` pattern already exists on input entries — consider reusing it or adding a `metricsOnly` flag.
- Do not change non-chat-mode behavior. `routeTarget === 'shell'` must continue rendering full responses in terminal.

### Acceptance Criteria
- [ ] Chat mode: terminal shows metrics line after LLM response, NOT full response text
- [ ] Chat mode: terminal shows spinner during API call
- [ ] Chat mode: terminal shows system messages and errors
- [ ] Chat mode: conversation history still works (entries still contain content for context building)
- [ ] Non-chat mode: full response renders in terminal (unchanged)
- [ ] Tests cover both chat-mode and non-chat-mode rendering
- [ ] All existing terminal tests still pass
