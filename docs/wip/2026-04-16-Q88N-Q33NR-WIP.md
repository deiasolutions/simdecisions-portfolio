### 1.1 [yellow] Phone → Factory spec submission flow — pick an MVP path

**What this is:** Q88N wants to submit specs from his phone. Flow: phone browser → chat interface → cloudnode (Railway) → hivenode (local PC) → `.deia/hive/queue/backlog/`. The chat piece "isn't working but should be." Two weeks ago we did a cloudnode audit that recommended seven relay specs; none were ever written.

**What the bee found (MOBILE-SUBMIT-001, Sonnet, ~$0.07):**

1. **The "chat primitive" is a red herring.** `ConversationPane.tsx` (291 lines) is display-only — it accepts a `messages` array and an `onCopy` callback, and that's it. No input box, no send button, no fetch calls, no service integration. It was built as a reference UI and never wired. When we said "chat isn't working but should be," we were pointing at the wrong component.
2. **The chat that actually works** is the terminal primitive in `chat.set.md` with `routeTarget: 'ai'`. It streams to `POST /api/llm/chat/stream` (SSE) on hivenode, backed by Claude via `anthropic.AsyncAnthropic`. This works on desktop against localhost and on mobile via the Vercel → Railway proxy in `vercel.json`.
3. **Spec submission already works on desktop.** `SpecSubmitForm` (in `browser/src/primitives/spec-submit/`) POSTs JSON to `http://127.0.0.1:8420/factory/spec-submit`. The hivenode endpoint in `factory_routes.py:348` validates with Gate 0 and writes to `.deia/hive/queue/backlog/SPEC-{TYPE}-{YYYYMMDD}-{HHMM}.md`. `acceptance_criteria` is a required field so generated specs don't fail Gate 0.
4. **No paste-a-spec-into-chat flow exists.** Terminal has no heuristic for "this message starts with `# SPEC:`, ask if the user wants to submit it." Submit is triggered only by explicit FAB tap or the `factory:open-spec-submit` bus event.
5. **No cloud → local relay exists.** Railway runs in `mode=cloud`, which disables queue_runner/scheduler/dispatcher and has no `.deia/` directory. The Railway node registry tracks `node_id → user_id` but has no `/factory/command` table, no WebSocket channel, no polling endpoint, no way to hand a spec to the local node.
6. **`/factory/spec-submit` on Railway is unprotected** — no `Depends(verify_jwt)`. Any caller can POST. On local hivenode that's fine; on Railway in cloud mode it's a hole, but also currently unreachable because the write target (`.deia/`) doesn't exist there.

**The MVP the bee recommends (four specs, ~8–10 bee-hours):**
- **CLOUDNODE-QUEUE-001** (haiku) — add `node_commands` table (id, node_id, command_type, payload JSON, status, timestamps) to Railway PostgreSQL.
- **CLOUDNODE-DISPATCH-001** (haiku) — JWT-protected `POST /factory/spec-submit` on Railway that looks up the user's online node via `node_store.discover(user_id)`, queues a `submit_spec` command, returns `{queueId, status: "queued"}`. Also `GET /factory/commands?node_id=` and `POST /factory/command-ack`.
- **CLOUDNODE-POLLING-001** (sonnet) — new `hivenode/factory_poller.py` background worker that polls Railway every 30s for pending commands, writes them to local `backlog/` through the existing Gate 0 code path, and ACKs back.
- **MOBILE-FRONTEND-001** (haiku) — have `SpecSubmitForm` pick its POST target based on `HIVENODE_URL` (localhost in dev, same-origin in production), and handle the new "queued for delivery" 202 response with a "delivered to your PC" toast.

**There's a cheaper alternative** (two specs, no cloud plumbing): detect that a terminal paste starts with `# SPEC:` / `## Priority`, prompt "Submit this to the factory?", and POST to the local endpoint we already have. Only works when the phone can reach localhost:8420 — i.e. same LAN or Tailscale, not over the open internet.

**Decision needed:** (A) MVP polling relay, (B) WebSocket relay (adds CLOUDNODE-RELAY-001 opus-sized), or (C) just chat-paste-detect for now and defer the cloud relay. Q33NR lean: **A**, because it unblocks phone submissions from anywhere and the 30s latency is fine for spec submit (this is not a chat-latency problem).

### 1.2 [yellow] Wiki vs Recon — pick a strategy

**What this is:** Two ideas converged today. First, the "recon-then-swarm" pattern — a single cheap/long-context bee reads the whole slice and writes a digest, downstream bees read the digest plus narrow files instead of each re-ingesting the full context. Second, Mr. AI came back on our proposal and said "you could avoid a lot of recon if you had a working wiki." So we dispatched WIKI-RECON-001 to audit what we have.

**What the bee found (WIKI-RECON-001, Sonnet, $4.62):**

- **Infrastructure is ~70% built.** `hivenode/wiki/` has working CRUD: 12 endpoints mounted at `/api/wiki/` (list, get, create, update, delete, search, tree, backlinks, etc.), a `wiki_pages` table with 19 columns plus a `wiki_edit_log` audit table, SQLite for local / PostgreSQL for Railway, a parser that extracts `[[wikilinks]]` and frontmatter, and four test files.
- **Frontend is 3-of-5 primitives functional.** WikiViewer + MarkdownViewer render pages, BacklinksPanel works, WikiTree shows the hierarchy via `tree-browser`. **WikiEditor** (in-UI page creation/editing) and **WikiSearch** (full-text + tag search) are both missing.
- **Content is 0%.** The bee inferred zero pages in the production DB (did not actually query — see §2.3 below). No ONET occupation pages, no tool taxonomy pages, no raw/ sources ingested.
- **Dispatch integration is 0%.** `grep` on `dispatch.py`, `run_queue.py`, and `.deia/config/injections/` returns no wiki references. The factory does not consult the wiki when assembling a bee's context.
- **Verdict on the wiki-vs-recon framing:** they are **complementary, not substitutes**. Six of seven proposed follow-on specs say `supersedes_recon_proposal: false`; one says `partial` (the dispatch-integration spec — a populated wiki genuinely would handle some lookups that a recon bee would otherwise redo).

**The three paths on the table:**
- **A. Both in parallel.** Ship recon-swarm now for immediate per-wave ROI (cheaper bulk bees, more consistent reads), seed the wiki on a slower track, wire wiki-into-dispatch when there's enough content to matter.
- **B. Wiki-first.** Invest in WIKI-SEED-001 + WIKI-SCHEMA-001 + WIKI-DISPATCH-001 before recon ships. Higher ceiling, longer time-to-value.
- **C. Recon-first.** Ship the ephemeral digest pattern now, revisit the wiki at later factory scale.

**Q33NR lean: A.** Matches the bee's implicit recommendation, decouples the two investments, and recon gives us wave-level wins while the wiki content is still being built.

**The seven wiki follow-on specs the bee sketched:**
- P0 (three of them): SEED-001 (seed ~10 ONET + 10-20 Tool Taxonomy pages), SCHEMA-001 (write `SCHEMA.md` and define `.wiki/` repo structure + page types), DISPATCH-001 (wire `dispatch.py` to inject relevant wiki pages into each bee's context based on `## Depends On` and area tags).
- P1: SEARCH-001 (full-text + tag search), EDITOR-001 (in-UI page editor).
- P2: PATHS-001 (fix a `wiki_root` path bug in `operations.py`), WATCHER-001 (directory watcher for `raw/` that auto-ingests).

**The three recon follow-on specs from this morning's proposal:**
- RECON-001 (research, sonnet) — digest format, staleness/invalidation rules, validation pass, fallback paths.
- RECON-002 (implementation, sonnet) — wire a recon step into `run_queue.py`, define digest file naming.
- RECON-003 (integration) — merge recon decision into the pick-bee skill (see §1.3).

**Decision needed:** pick a path (A/B/C), then green-light the corresponding spec set.

### 1.3 [yellow] Pick-Bee skill — when should which model run a task?

**What this is:** Today's factory picks bees by whatever the spec says in `## Model Assignment`, which is usually whatever the spec author guessed. We discussed making that decision data-driven. The proposal — sent to Mr. AI for a second opinion — has two parts.

**Part A: the skill itself (`.claude/skills/pick-bee.md`).** Natural-language policy that Q33N consults before dispatch. Seven axes: task archetype (research / CRUD / architecture / refactor / bugfix / test / UI), estimated token scope (spec + files-to-read length), spec quality signals (Gate 0 density, AC specificity), historical performance for that archetype/model combo, priority × budget, vendor capability table (haiku good at string-juggling, sonnet good at most things, opus good at architecture, gemini cheap long-context for recon, openai TBD), and diversity hedge (don't let one vendor monopolize a wave). The skill reads this JSON.

**Three follow-on specs if Q88N greenlights:**

- PICKBEE-003 (implementation, sonnet + haiku) — write the skill, write the helper CLI, dogfood on 10 specs, tune thresholds.

**Data we don't have yet but need:** `## Archetype` field on specs (or inferred from title/content), `estimated_tokens_at_dispatch` captured in the ledger, an outcome quality label beyond "completed/failed," and a human-verdict column Q88N can fill in post-hoc when a bee picks wrong.
